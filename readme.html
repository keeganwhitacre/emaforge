Overview & Architecture

EMA Forge is an open-source, fully serverless toolkit designed for researchers conducting Ecological Momentary Assessment (EMA) and Digital Phenotyping studies.

Traditional EMA tools require setting up a backend server, configuring a database, managing user accounts, and navigating complex HIPAA/IRB compliance requirements regarding data transmission in transit. EMA Forge takes a completely different approach.

Core Concept: Your study is compiled into a single, static web app. Participants complete the study in their mobile browser. The data is generated, stored, and downloaded entirely on their device.

Why Serverless?

By eliminating the server, EMA Forge radically simplifies the IRB approval process and technical overhead:

  • Zero Backend Setup: Host your study for free on GitHub Pages, Vercel, or any static file server.
  • Ultimate Privacy: No data is ever transmitted to a central server. Participants physically download their data or email it to the research team at the end of the study phase.
  • Guaranteed Offline Capability: Once loaded in the browser, the study can function even if the participant loses cell service during a task.

Building the Study

The Study Builder (builder.html) is your primary authoring environment. It operates entirely in your browser.

  1. Study: Define global settings, colors, and crash-recovery rules.
  2. Questions: Construct your survey battery. You can insert page breaks to group questions and apply conditional logic.
  3. Schedule: Define the longitudinal nature of your study. Set days, daily prompt counts, and specific time windows.
  4. Tasks: Enable advanced cognitive or physiological tasks (e.g., ePAT, Camera HR) to be inserted into specific phases.

As you build, the config.json file is generated under the hood. You can view this live by clicking the JSON button in the topbar.

Export & Deployment

When your study is ready, click Export. You will be given two choices:

  • Single HTML File: A perfectly self-contained artifact. Every piece of logic, CSS, and styling is minified into one file. Easiest for quick pilots or emailing directly to a participant.
  • Static-hosting Bundle (Recommended): A `.zip` containing your `index.html`, assets, and `config.json`. Extract this folder and upload it to GitHub Pages. If you need to fix a typo in a question, just edit the `config.json` directly—no need to re-export!

Participant Routing (URLs)

Since there is no server to "assign" a participant a prompt, EMA Forge uses URL Query Parameters to route participants to the correct session.

If your study is hosted at https://your-lab.edu/study/, a link texted to a participant would look like this:

https://your-lab.edu/study/?id=104&day=2&session=morning
Parameter Description
id Unique participant identifier (e.g., 104)
day The current study day (e.g., 1 to 14)
session Matches the window name defined in the Schedule tab (e.g., morning)

Pro-tip: Use the Deployment tab in the Builder. Provide your base URL, and EMA Forge will generate a massive CSV containing every single pre-formatted link for every participant, day, and session. You can drop this CSV straight into Qualtrics, Twilio, or your SMS scheduler.

Data Formats

When a participant finishes a session, their device automatically downloads the data. Depending on your configuration, this will be:

  • CSV (Default): One row per question answered. Extremely easy to concatenate across participants using R or Python.
  • JSON: Included automatically if the session contained high-density signal data (like the PPG camera feed or ePAT reaction times).

Files are automatically named securely: ema_data_[ID]_[DAY]_[SESSION].csv

Using the Dashboard

The Data Dashboard (dashboard.html) is your local command center.

Because EMA Forge has no database, you use the Dashboard by clicking Import Folder and selecting a local folder on your computer that contains all the collected participant JSON/CSV files. The dashboard runs local parsing algorithms to instantly calculate:

  • Overall and per-participant compliance rates.
  • Signal noise and invalid response detection (e.g., speeding through questions <30s).
  • Latency (time between receiving a link and opening it).

Crash Recovery & Protection

By default, EMA Forge enables two vital protections in the wild:

  1. Session Locking: A participant cannot accidentally submit the day=2&session=morning link twice. If they click it again, they see a "Session Complete" screen. (Can be overridden by researchers using ?force=1).
  2. Crash Recovery: If a participant accidentally closes Safari/Chrome halfway through a session, reopening the link instantly restores their progress. Completed modules are saved in local storage.

WebRTC PPG Capture

EMA Forge includes a module to capture heart rate data using a smartphone camera (Photoplethysmography). It asks the user to place their finger over the rear camera and flashlight.

The module runs local WebRTC processing and bandpass filtering, extracting the color-intensity variance to map the blood pulse wave. This raw signal data is exported in the JSON payload.

ePAT Integration

The Ecological Phase Adjustment Task (ePAT) is built natively into EMA Forge. If enabled in the Tasks tab, you can slot it into a phase sequence (e.g., complete the survey, then do the ePAT, then complete a post-survey).

The ePAT engine handles high-precision AudioContext scheduling to ensure stimulus timing remains perfectly accurate across different mobile browsers.
